# Деплой на Railway 🚂

Railway — це чудова платформа для деплою як бекенду, так і фронтенду. У вас є два основні варіанти: деплоїти їх разом (Monorepo) або окремо (як два різні сервіси).

## Варіант 1: Деплой окремо (Рекомендовано)

Деплой бекенду та фронтенду як двох окремих сервісів у Railway дає більше гнучкості, дозволяє окремо масштабувати їх і спрощує процес збірки.

### Крок 1. Підготовка репозиторію

Переконайтеся, що ваш код (з Dockerfile-ами, які ми щойно створили) запущено на GitHub.

### Крок 2. Деплой Бекенду (FastAPI)

1. Увійдіть в [Railway](https://railway.app/).
2. Натисніть **New Project** -> **Deploy from GitHub repo**.
3. Виберіть ваш репозиторій.
4. Після додавання репозиторію, перейдіть до налаштувань створеного сервісу (це буде перший сервіс).
5. Знайдіть **Root Directory** в налаштуваннях (Settings -> Service) і вкажіть `/backend`.
6. Railway автоматично знайде `Dockerfile` у папці `/backend` і збере образ.
7. Перейдіть у вкладку **Variables** та додайте всі необхідні змінні середовища (наприклад, `OG_PRIVATE_KEY` та інші, що були у вашому `.env`).
8. У вкладці **Settings** -> **Networking** натисніть **Generate Domain**, щоб отримати публічний URL для вашого бекенду. (Збережіть його, він знадобиться для фронтенду).

### Крок 3. Деплой Фронтенду (Vite + React)

Для продакшену фронтенду краще використовувати статичний хостинг. Ми можемо зробити це у тому ж проєкті Railway.

1. У тому ж проєкті Railway натисніть **New** (або плюсик `+` зверху) -> **GitHub Repo** і знову виберіть той самий репозиторій.
2. Перейдіть до налаштувань нового сервісу.
3. Змініть **Root Directory** на `/frontend`.
4. Оскільки це Vite (React) додаток для продакшену, **не обов'язково** використовувати Docker. Railway має нативний білдер (Nixpacks), який автоматично розпізнає `package.json`.
5. Railway може запропонувати автоматично запустити `npm run build` і роздавати статику. Якщо ви хочете використовувати Docker для фронтенду на Railway, замініть локальний `Dockerfile` для фронтенду на такий (multi-stage build), проте автоматичного Nixpacks зазвичай достатньо:

   ```dockerfile
   # Приклад Production Dockerfile для Frontend
   FROM node:20-alpine AS builder
   WORKDIR /app
   COPY package*.json ./
   RUN npm install
   COPY . .
   RUN npm run build

   FROM nginx:alpine
   COPY --from=builder /app/dist /usr/share/nginx/html
   EXPOSE 80
   CMD ["nginx", "-g", "daemon off;"]
   ```

6. У вкладці **Variables** додайте змінну `VITE_API_URL` і вкажіть публічний домен вашого бекенду (з Кроку 2). Це дуже важливо, щоб фронтенд знав, куди робити запити у продакшені!
7. Згенеруйте публічний домен (Generate Domain) у вкладці Networking.
8. Обов'язково додайте цей домен фронтенду у список `allow_origins` (CORS) у `backend/app/main.py` або у відповідну змінну середовища бекенду `FRONTEND_ORIGIN`, якщо вона там підтримується (бачу у вас `settings.frontend_origin`).

## Варіант 2: Деплой разом (Monorepo Docker-based)

Якщо ви хочете запустити все в одному контейнері (або використовуючи `docker-compose` на Railway, що частково підтримується, або зібравши фронтенд і роздаючи його через FastAPI), це менш зручно, але можливо. Проте, Railway ідеально підходить для мікросервісів, тому **набагато краще розгорнути 2 сервіси в 1 проєкті**, як описано у Варіанті 1.

Railway дозволяє створювати так звані "Services" всередині одного "Project", що по суті працює як docker-compose, де сервіси можуть спілкуватися один з одним по внутрішній мережі (Private Networking), але у нашому випадку фронтенд (браузер) все одно повинен робити запити на публічний URL бекенду, тому потрібен публічний домен для обох.

### Резюме по локальному запуску:

Тепер, маючи `docker-compose.yml`, для локального запуску вам достатньо лише виконати:

```bash
docker-compose up --build
```

Бекенд буде доступний на `http://localhost:8000`, а фронтенд на `http://localhost:5173`. Фронтенд автоматично проксіює звернення на `/api` до бекенду через налаштування Vite.
